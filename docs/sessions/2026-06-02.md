# 세션: 2026-06-02

## 주제: 나라장터 v1.0 필터 설계 — API 검증

프로토타입 성공 후 본격 개발(v1.0)로 전환. 나라장터 단일 사이트에 집중하기로 결정.
필터 방식은 "수집 후 거르기"가 아닌 **"필터 후 제한적 수집"** 방향으로 전환.
(이유: 기존 `output/나라장터`에 전량 수집으로 파일이 폭증, 컷오버 발생)

## 사용자 확정 사항

- **v1.0 범위:** 나라장터만. 다른 4개 사이트(NIPA/NIA/MSS/ETRI) 기능 개발 보류
- **키워드 입력:** `config.yaml`에서 수정, 복수 인자 가능 구조
- **필터 방식:** 3번(필드별 타겟 키워드). 단, **발주기관은 필터 조건에서 제외**
- **단계구분:** 입찰공고 + 사전규격 **둘 다 포함**

## API 검증 결과 (실제 호출로 확인)

### 단계구분 — 동일 인증키로 둘 다 접근 가능

| 단계 | 서비스 경로 |
|---|---|
| 입찰공고 | `https://apis.data.go.kr/1230000/ad/BidPublicInfoService` |
| 사전규격 | `https://apis.data.go.kr/1230000/ao/HrcspSsstndrdInfoService` (경로가 `/ad/`가 아니라 `/ao/`) |

### 입찰공고 — 검색어 서버 필터 가능 (핵심)

- 현재 코드가 쓰던 `getBidPblancListInfoServc`는 **날짜 범위 전량 조회**라 키워드 무시됨 (17,393건 그대로)
- **`getBidPblancListInfoServcPPSSrch`** ("나라장터 검색조건에 의한 입찰공고 용역 조회")는 `bidNtceNm` 파라미터로 **서버 검색 작동** (인공지능 → 52건, 공고명 정확 매칭)
- → **v1.0은 PPSSrch 오퍼레이션으로 전환 필요**

### 입찰공고 — 계약방법/낙찰방법

- 응답 필드로 존재: `cntrctCnclsMthdNm`(계약방법), `sucsfbidMthdNm`(낙찰방법)
- 검색 파라미터로는 **작동 안 함** → 로컬 필터로만 가능
- 실제 값:
  - 계약방법: 일반경쟁 / 제한경쟁 / 수의계약
  - 낙찰방법: 협상에의한계약 / 적격심사제 / 소액수의견적 / 규격가격동시입찰 등

### 사전규격 — 검색어 서버 필터 가능 (PPSSrch 변형)

- ⚠️ 정정: 처음 테스트한 `getPublicPrcureThngInfoServc`(비-PPSSrch)는 키워드 무시 (15,424건 그대로)
- **`getPublicPrcureThngInfoServcPPSSrch`** ("나라장터 검색조건" 변형)는 `prdctClsfcNoNm` 파라미터로 **서버 검색 작동** (인공지능 → 57건, 품명 정확 매칭)
- 품명 필드: `prdctClsfcNoNm`
- → 입찰공고와 동일하게 **검색어 API 전달 → 제한적 수집** 가능

## 설계 결론 (수집 방식)

- **입찰공고:** 검색어를 API(`getBidPblancListInfoServcPPSSrch`, `bidNtceNm`)에 전달해 소수만 수집 → 계약방법·낙찰방법은 로컬 필터
- **사전규격:** 검색어를 API(`getPublicPrcureThngInfoServcPPSSrch`, `prdctClsfcNoNm`)에 전달해 소수만 수집 (입찰공고와 동일 패턴)
- **공통 원칙:** 첨부파일 다운로드는 **필터 통과 건에 대해서만** 수행 (목록 조회는 가볍고, output 폭증의 주범은 첨부파일 다운로드)
- **복수 키워드:** 두 단계 모두 키워드당 PPSSrch 1회 호출 후 합쳐서 중복 제거 (검색어 파라미터는 한 번에 1개로 추정)

## 확정된 필터 스펙 (Planner, 2026-06-02 확정)

### config.yaml 형식 (sites.g2b 확장)

```yaml
sites:
  g2b:
    stages:
      bid_notice:   true   # 입찰공고
      pre_standard: true   # 사전규격
    date_range_days: 3
    search_keywords:       # ★ 서버 검색(OR), 최소 1개 필수
      - "인공지능"
      - "AI"
      - "데이터"
    exclude_keywords:      # 로컬, 검색어보다 우선
      - "청소"
      - "경비"
    contract_methods: []   # 로컬, 입찰공고만. 값: 일반경쟁/제한경쟁/수의계약
    award_methods:         # 로컬, 입찰공고만, 부분일치
      - "협상에의한계약"
    budget_min: null       # 로컬, 원 단위, null=제한없음
    budget_max: null
```

### 필터 적용 순서 (AND 결합, search_keywords 내부만 OR)
1. search_keywords — 서버(API), 단어별 검색→합침→중복제거
2. exclude_keywords — 로컬, 공고명/품명 매칭 시 제외 (1번보다 우선)
3. contract_methods — 로컬(입찰공고만), 비면 통과
4. award_methods — 로컬(입찰공고만), 부분일치, 비면 통과
5. budget_min/max — 로컬, 범위 밖 제외

- **첨부파일 다운로드는 5단계 모두 통과한 공고만** (output 폭증 방지 핵심)
- **사전규격은 계약방법·낙찰방법 필드가 없어 3·4번 미적용(통과)**

### 확정된 열린 질문 (모두 권고안 채택)
1. 검색어 비면 → **실행 거부 + 경고** (최소 1개 필수)
2. 낙찰방법 → **앞부분 분류명 부분일치**
3. 검색어 AND 조합 → v1.0 제외, **OR만** (AND는 v1.1)
4. 계약방법/낙찰방법이 사전규격에 없는 건 → **그대로 수용**

### 마이그레이션
- 전역 `filters.keywords` → `sites.g2b.search_keywords` (의미: 로컬→서버검색)
- `filters.match_logic`, `filters.categories` → 제거
- `sites.g2b.api_base` → architect 단계에서 PPSSrch 2개 오퍼레이션 경로로 교체

- **분석 트리거:** 구독 유지 — 버튼은 다운로드+변환(분석 준비)까지, 실제 분석은 Claude Code rfp-analyzer로 트리거 (유료 API 미도입)

### 첨부파일 형식 실측 (다운로드된 파일 기준)
