# PRD: Public Institution Announcement Scraper

## 1. Introduction / Overview

주요 공공기관 및 정부 포털에서는 사업 공고를 수시로 게재한다. 이를 매일 수동으로 확인하는 것은 비효율적이며 누락 위험이 있다.

이 도구는 지정된 5개 공공기관 웹사이트를 자동으로 스크랩하여, 사용자가 설정한 키워드와 카테고리에 해당하는 공고만 추려낸 뒤 Excel 파일로 정리하고, 각 공고의 **첨부파일(PDF, HWP 등)도 함께 다운로드**한다. 사용자는 매일 아침 Excel 파일 하나만 열어 오늘의 관련 공고를 한눈에 파악하고, 첨부 파일도 바로 확인할 수 있다.

---

## 2. Goals

- 지정된 5개 공공기관 사이트의 사업 공고를 매일 자동 수집한다.
- 키워드 + 카테고리 조합으로 필터링하여 관련 공고만 추출한다.
- 공고 핵심 정보를 Excel 파일에 정리하고, 기존 이력은 유지하며 신규 공고만 추가한다.
- 공고별 첨부파일(공고문, 제안요청서 등)을 자동으로 다운로드하여 로컬에 저장한다.
- 중복 공고가 파일에 쌓이지 않도록 방지한다.

---

## 3. User Stories

- **사업자로서**, 매일 아침 스크립트가 자동 실행되어 오늘 새로 올라온 공고 목록이 Excel에 정리되어 있기를 원한다.
- **사용자로서**, 내가 관심 있는 키워드(예: "AI", "데이터", "용역")만 걸러진 공고를 보고 싶다.
- **사용자로서**, Excel 각 행에서 공고 제목, 기관명, 마감일, 예산, 요약, 링크를 한눈에 확인하고 싶다.
- **사용자로서**, 공고에 딸린 첨부파일(공고문, RFP 등)이 자동으로 다운로드되어 있어 바로 열람하고 싶다.
- **사용자로서**, 이전에 수집된 공고 이력이 보존되고 새 공고만 추가되기를 원한다.

## 4. Functional Requirements

### 4.1 Target Sources

1. 시스템은 아래 5개 사이트에서 공고를 수집**해야 한다**:

   | # | 기관명 | URL | 비고 |
   |---|---|---|---|
   | 1 | 중소벤처기업부 | https://www.mss.go.kr | 공고/입찰 게시판 |
   | 2 | 나라장터 (G2B) | https://www.g2b.go.kr | 공공데이터포털 API 우선 사용 |
   | 3 | 정보통신산업진흥원 (NIPA) | https://www.nipa.kr | 사업공고 게시판 |
   | 4 | 한국지능정보사회진흥원 (NIA) | https://www.nia.or.kr | 공고 게시판 |
   | 5 | 한국전자통신연구원 (ETRI) | https://ebid.etri.re.kr | 입찰공고 게시판 |

2. 각 소스는 `config.yaml`에서 개별적으로 **활성화/비활성화** 할 수 있어야 한다.

### 4.2 Filtering

3. 시스템은 **키워드 목록**(예: `["AI", "데이터", "용역", "소프트웨어", "개발"]`)을 받아, 공고 제목 또는 내용에 하나 이상의 키워드가 포함된 공고를 필터링**해야 한다**.
4. 시스템은 **카테고리 코드 목록**(나라장터의 분류 코드 등)을 받아 해당 카테고리의 공고를 필터링**해야 한다**.
5. 필터 논리는 설정 가능**해야 한다**:
   - 기본값: 키워드 OR 카테고리 중 하나라도 일치하면 수집.
6. 키워드, 카테고리, 기타 모든 설정은 소스코드를 수정하지 않고 `config.yaml` 하나만 편집하여 변경할 수 있어야 한다.
7. 필터를 모두 비워두면 해당 사이트의 전체 공고를 수집**해야 한다** (필터 없음 = 전체 수집).

### 4.3 Data Extraction (Excel 컬럼)

8. 매칭된 각 공고에 대해 다음 필드를 추출하고 저장**해야 한다**:

   | 컬럼명 | 설명 |
   |---|---|
   | 공고번호 | 해당 사이트의 고유 공고 ID |
   | 공고명 | 공고 제목 |
   | 발주기관 | 발주 기관명 |
   | 출처사이트 | 수집된 사이트명 (예: "나라장터", "NIPA") |
   | 공고일 | 공고 게재일 |
   | 마감일시 | 제출 마감일시 |
   | 예산금액 | 사업 예산 (원화, KRW) |
   | 내용요약 | 공고 내용의 앞부분 텍스트 (최대 300자) |
   | 첨부파일 수 | 다운로드된 첨부파일 개수 |
   | 첨부파일 경로 | 로컬에 저장된 첨부파일 폴더 경로 |
   | 공고 링크 | 공고 상세 페이지 URL |
   | 수집일시 | 이 행이 스크랩된 날짜/시각 |

### 4.4 Attachment Download (첨부파일 다운로드)

9. 시스템은 각 공고 상세 페이지에서 **첨부파일 링크를 탐지**하고 모두 다운로드**해야 한다**.
10. 지원 파일 형식: `.pdf`, `.hwp`, `.hwpx`, `.docx`, `.xlsx`, `.zip`, `.pptx`
    - 위 형식 이외의 파일은 다운로드하지 않고 로그에 기록한다.
11. 첨부파일은 다음 폴더 구조로 저장**해야 한다**:
    ```
    output/
    └── attachments/
        └── {출처사이트}/
            └── {공고번호}/
                ├── 공고문.pdf
                └── 제안요청서.hwp
    ```
12. 동일한 공고번호의 첨부파일이 이미 존재하면 **재다운로드하지 않고 건너뛴다** (중복 방지).
13. 다운로드 실패 시(네트워크 오류, 접근 제한 등) 해당 파일은 건너뛰고 로그에 기록하며, 나머지 공고 처리를 계속**해야 한다**.
14. Excel의 `첨부파일 경로` 컬럼에는 해당 폴더의 **절대 경로**를 기록하여, 사용자가 클릭으로 폴더를 열 수 있도록 한다.

### 4.5 Excel Output

15. 결과를 `config.yaml`에 지정된 출력 폴더에 `.xlsx` 파일로 저장**해야 한다**.
16. 파일명 규칙은 설정 가능**해야 한다**:
    - `rolling_file: true` → 단일 파일(`announcements.xlsx`)에 누적 추가
    - `rolling_file: false` → 실행 날짜별 파일(`announcements_2026-02-24.xlsx`) 생성
17. 중복 제거 기준: `공고번호` + `출처사이트` 조합이 이미 파일에 존재하면 추가하지 않는다.
18. 신규 공고는 기존 파일 아래에 **추가(append)**하며 기존 데이터를 덮어쓰지 않는다.
19. Excel 파일은 헤더 행을 포함하고 열 너비를 자동 조정**해야 한다**.
20. **마감일이 7일 이내인 행**은 배경색을 주황색으로 표시**해야 한다** (긴급 표시).
21. `공고 링크`와 `첨부파일 경로` 컬럼은 **클릭 가능한 하이퍼링크**로 삽입**해야 한다**.

### 4.6 Scheduling (자동 실행)

22. 시스템은 Python `schedule` 라이브러리 또는 Windows 작업 스케줄러를 통한 자동 실행을 지원**해야 한다**.
23. 권장 방식: **Windows 작업 스케줄러**로 `python main.py --once`를 매일 지정 시각에 실행 (Python 프로세스를 24시간 상시 구동하는 것보다 안정적).
24. 실행 시각은 `config.yaml`에서 설정 가능**해야 한다** (기본값: 매일 08:00).
25. 매 실행마다 로그 파일에 다음을 기록**해야 한다**: 시작 시각, 수집된 공고 수, 신규 추가 공고 수, 다운로드된 첨부파일 수, 발생한 오류.

### 4.7 Error Handling

26. 특정 사이트에 접근 불가하거나 응답이 예상과 다를 경우, 해당 사이트 오류를 로그에 기록하고 나머지 사이트 처리를 계속**해야 한다** (fail-safe).
27. 공고의 특정 필드(예: 예산금액)가 없을 경우 빈 셀로 저장하고 스크립트를 중단하지 않는다.
28. 각 요청 사이에 1~2초 딜레이를 두어 사이트 차단을 방지**해야 한다**.
29. `User-Agent` 헤더를 일반 브라우저처럼 설정하여 기본적인 봇 차단을 우회**해야 한다**.

## 5. Non-Goals (Out of Scope)

- **UI/웹 대시보드 없음** — 커맨드라인 스크립트 형태로만 제공.
- **자동 입찰/신청 없음** — 공고 정보 수집 및 파일 다운로드만 수행.
- **로그인 인증 없음** — 로그인 없이 접근 가능한 공개 공고만 수집. (로그인이 필요한 페이지는 이번 버전 범위 외)
- **이메일/푸시 알림 없음** — 출력물은 Excel 파일과 로컬 첨부파일뿐.
- **AI 요약 없음** — `내용요약` 필드는 공고 페이지의 텍스트 앞부분(최대 300자)을 그대로 사용.
- **압축파일 내부 파싱 없음** — `.zip` 파일은 다운로드만 하고 압축 해제 및 내부 파일 분석은 하지 않음.

## 6. Design Considerations

### config.yaml 구조 (예시)

```yaml
sources:
  mss:        true    # 중소벤처기업부
  g2b:        true    # 나라장터
  nipa:       true    # 정보통신산업진흥원
  nia:        true    # 한국지능정보사회진흥원
  etri:       true    # 한국전자통신연구원

filters:
  keywords: ["AI", "데이터", "용역", "소프트웨어", "개발", "정보화"]
  categories: []         # 나라장터 업종코드. 비워두면 전체 수집
  match_logic: "OR"      # OR | AND

attachments:
  enabled: true
  download_dir: "./output/attachments"
  allowed_extensions: [".pdf", ".hwp", ".hwpx", ".docx", ".xlsx", ".zip", ".pptx"]
  max_file_size_mb: 50   # 이 크기 초과 파일은 건너뜀

output:
  directory: "./output"
  rolling_file: true
  filename: "announcements.xlsx"

schedule:
  enabled: true
  time: "08:00"          # 매일 실행 시각 (24시간 형식)

api_keys:
  g2b_api_key: "YOUR_DATA_GO_KR_API_KEY"   # 공공데이터포털 발급 키
```


## 7. Technical Considerations

- **언어:** Python 3.10+
- **핵심 라이브러리:**

  | 라이브러리 | 용도 |
  |---|---|
  | `requests` | HTTP 요청 |
  | `BeautifulSoup4` | HTML 파싱 |
  | `pandas` + `openpyxl` | Excel 파일 생성 및 서식 |
  | `PyYAML` | config.yaml 파싱 |
  | `schedule` | 인-프로세스 스케줄링 (옵션) |
  | `logging` | 실행 로그 기록 |
  | `pathlib` | 파일 경로 관리 |
  | `tqdm` | 다운로드 진행 표시 (선택) |

- **나라장터 API 우선 사용:**
  - 공공데이터포털 오픈API: `https://apis.data.go.kr/1230000/BidPublicInfoService04`
  - HTML 스크래핑보다 안정적이므로 최우선 사용.
  - API 키는 [공공데이터포털](https://www.data.go.kr)에서 무료 발급 가능.

- **사이트별 스크래핑 주의사항:**
  - **중소벤처기업부 (mss.go.kr):** 게시판 목록 → 상세 페이지 → 첨부파일 순서로 크롤링.
  - **NIPA (nipa.kr):** 사업공고 게시판, 페이지네이션 처리 필요.
  - **NIA (nia.or.kr):** 공고 게시판, JavaScript 렌더링 여부 확인 필요 — 필요 시 `selenium` 또는 `playwright` 추가 고려.
  - **ETRI (ebid.etri.re.kr):** 전자입찰 시스템, 공개 목록 페이지에서 수집.

- **첨부파일 다운로드 전략:**
  - `Content-Disposition` 헤더에서 파일명 추출, 없으면 URL 끝에서 추출.
  - 한글 파일명은 `urllib.parse.unquote`로 디코딩.
  - 파일 크기가 `max_file_size_mb` 초과 시 건너뜀.

- **요청 간 딜레이:** 사이트별 1~3초 랜덤 딜레이 (`time.sleep(random.uniform(1, 3))`).

## 8. Success Metrics

- 7일 연속 스케줄 실행 시 오류 없이 정상 완료.
- 각 사이트의 키워드 매칭 공고가 100% 수집됨 (수동 3회 이상 비교 검증).
- Excel 파일에 중복 행이 0건.
- 각 공고의 첨부파일이 지정 폴더에 정상 저장되고 열람 가능.
- Excel의 하이퍼링크 클릭 시 브라우저/파일 탐색기가 정상 열림.

## 9. Open Questions

1. **나라장터 API 키:** 공공데이터포털 API 키가 이미 있는가? 없다면 발급 절차를 README에 포함해야 한다.
2. **NIA JavaScript 렌더링:** nia.or.kr의 공고 목록이 JavaScript로 동적 렌더링되는 경우, `requests`만으로 수집이 불가할 수 있다. 이 경우 `playwright`(Headless 브라우저) 도입 여부를 결정해야 한다.
3. **첨부파일 용량 제한:** `max_file_size_mb` 기본값을 얼마로 설정할 것인가? (현재 PRD 예시: 50MB)
4. **로그 보존 기간:** 오래된 로그 파일은 자동으로 삭제할 것인가? (예: 30일 이후 자동 삭제)
5. **ETRI 로그인:** ebid.etri.re.kr 일부 상세 페이지가 로그인을 요구할 경우, 공개 요약 정보만 수집하는 것으로 범위를 제한할 것인가?
